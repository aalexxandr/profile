# Стандарты кода

Цель — понятный, структурированный и красивый код, который легко читать через месяц и безопасно менять агентам.

## TypeScript

- Пиши строгие типы для props, словарей, config и helper-функций.
- Не используй `any`, кроме случаев интеграции с нетипизированным внешним API; тогда сузь тип как можно раньше.
- Предпочитай `type` для props и простых структур.
- Не дублируй типы словаря вручную: используй `Dictionary`, `Locale`, `PageKey` из i18n.

## React и Next.js

- Серверные компоненты — режим по умолчанию.
- `"use client"` добавляй только там, где есть hooks, browser API, event handlers, animation refs или client-only библиотеки.
- Не поднимай client boundary на страницу или layout, если можно оставить его на маленьком компоненте.
- Next.js-специфичные решения сверяй с `node_modules/next/dist/docs/`.

## Компоненты и композиция

- Делить все крупные UI-блоки на компоненты с одной понятной ответственностью.
- Если JSX становится длинным, вложенным или содержит несколько смысловых зон, выдели подкомпоненты рядом с текущим файлом.
- Нетривиальную логику выноси из JSX:
  - reusable client logic — в hook `useSomething`;
  - pure calculation/formatting — в helper;
  - slice-specific helper — в место, выбранное по skill `feature-sliced-design`;
  - shared helper — в `src/shared/lib`, если он не знает о домене.
- JSX должен описывать структуру интерфейса, а не хранить сценарную логику.
- Локальные подкомпоненты можно держать в том же файле, пока файл остается читаемым. Если компонент начинает жить своей жизнью, вынеси его в отдельный файл.

## UI и Tailwind

- Tailwind-классы должны быть читаемыми и стабильными.
- Для условных классов используй `clsx`.
- Не собирай классы динамическими строками, если это ломает анализ Tailwind/Prettier.
- Сохраняй текущую визуальную стилистику: аккуратные pixel/retro детали, адаптивные размеры, локальные assets.
- Всегда проверяй, что текст не ломает контейнер на мобильных и desktop viewport.

## Accessibility и i18n

- Интерактивные элементы должны иметь понятный accessible name.
- Изображения получают осмысленный `alt` из словаря; декоративные элементы должны быть явно декоративными, если они не несут смысла.
- Все пользовательские строки добавляй в `ru` и `en` словари.
- Metadata страниц создавай через существующие i18n helpers.

## Файлы

- Держи файлы небольшими и сфокусированными.
- Не смешивай UI, data fetching, formatting и state machine в одном большом компоненте.
- Не создавай абстракцию без повторного использования или явного упрощения.
- Следуй соседним паттернам, прежде чем вводить новый.
